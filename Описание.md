## 6. Требования к тестированию (общие для всех типов)

### 6.1 Методы тестирования
Выбрать **минимум 4 метода** из таблицы (виды по 8 категориям):

1. По принципам работы: позитивное / негативное  
2. По уровню функционального: smoke / critical path / extended  
3. По исполнителям: альфа / бета  
4. По целям: функциональное / нефункциональное (производительность, нагрузка, масштабируемость, объёмное, стресс, инсталляционное, GUI/UI, usability, локализация, безопасность, надёжность, регрессионное, повторное)  
5. По запуску кода: статическое / динамическое  
6. По доступу к коду: белый / серый / чёрный ящик  
7. По уровню детализации: модульное / интеграционное / системное / приёмочное  
8. По автоматизации: ручное / автоматическое  

### 6.2 Тест-план
Документ, содержащий:
- Идентификатор, введение
- Объект тестирования
- Функции: будут / не будут тестироваться
- Подходы
- Критерии прохождения / приостановления
- Результаты, задачи
- Ресурсы, обязанности, роли
- Расписание, риски, согласования

### 6.3 Тест-кейсы
Каждый тест-кейс включает:
- **PreConditions**
- **Steps**
- **Expected result**

### 6.4 Баг-репорты
Обязательные атрибуты:
- **Серьёзность** (S1–S5: Blocker, Critical, Major, Minor, Trivial)
- **Приоритет** (высокий / средний / низкий)

### 6.5 Отладка
Проводится для разработанной системы.

## 7. Ключевые замечания и ограничения

- **Запрещена генерация случайных данных** для ML.
- **Запрещено использование CMS** для веб-ресурса (если тема не требует явно).
- **Запрещены кроссплатформенные фреймворки** для мобильной разработки.
- **Библиотеки** — только последние стабильные версии, без покупки токенов/аккаунтов.
- **Отсутствие среды проверки модели** → нарушение сдачи.
- **Сборка под Linux** обязательна для игр.
- **Все файлы БД** должны быть в репозитории (для SQL-решений).
