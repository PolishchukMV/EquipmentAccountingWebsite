# Тест-план системы учёта оборудования

## 1. Введение

### 1.1 Назначение документа
Документ описывает стратегию тестирования системы учёта оборудования (Equipment Accounting System), разработанной для ООО «ПроИнфоСервис». Тест-план определяет scope, подходы, ресурсы и расписание тестирования.

### 1.2 Область применения
Тестирование охватывает веб-приложение на Django 5.0.6, включая:
- Модели данных и бизнес-логика
- Представления (views) и маршруты (URL)
- Формы валидации
- Экспорт данных (Excel, Word)
- Безопасность и авторизация

### 1.3 Ссылки
- Требования: `Описание.md`
- Документация API: `REQUIREMENTS.md`
- Отчёт о соответствии: `CORRESPONDENCE_REPORT.md`
- Документ по безопасности: `SECURITY.md`

## 4. Подходы к тестированию

### 4.1 Методы тестирования

#### 4.1.1 По принципам работы
| Метод | Описание | Пример |
|-------|----------|--------|
| **Позитивное тестирование** | Проверка корректной работы при валидных данных | Создание оборудования с корректными полями |
| **Негативное тестирование** | Проверка обработки ошибок и невалидных данных | Ввод отрицательной цены, дубликат инвентарного номера |

#### 4.1.2 По уровню функционального тестирования
| Метод | Описание | Пример |
|-------|----------|--------|
| **Smoke тестирование** | Проверка критических функций | Вход в систему, создание оборудования |
| **Critical Path** | Проверка основных сценариев | CRUD операций для оборудования |
| **Extended тестирование** | Полное покрытие функционала | Все формы, все views, все модели |

#### 4.1.3 По целям тестирования
| Метод | Описание | Пример |
|-------|----------|--------|
| **Функциональное** | Проверка соответствия требованиям | Все CRUD операции |
| **Безопасность** | Проверка защиты данных | CSRF, XSS, права доступа |
| **UI/UX** | Проверка интерфейса | Адаптивность, навигация |
| **Регрессионное** | Проверка после изменений | После каждого коммита |

#### 4.1.4 По запуску кода
| Метод | Описание | Пример |
|-------|----------|--------|
| **Статическое** | Анализ кода без запуска | Проверка линтером (PEP8) |
| **Динамическое** | Тестирование работающей системы | unittest, pytest |

#### 4.1.5 По доступу к коду
| Метод | Описание | Пример |
|-------|----------|--------|
| **Белый ящик** | Тестирование с доступом к коду | Модульные тесты моделей |
| **Чёрный ящик** | Тестирование без знания внутренней структуры | UI-тесты через браузер |

#### 4.1.6 По уровню детализации
| Метод | Описание | Пример |
|-------|----------|--------|
| **Модульное** | Тестирование отдельных компонентов | Тесты моделей, тесты форм |
| **Интеграционное** | Проверка взаимодействия компонентов | Тесты views, тесты URL |
| **Системное** | Тестирование всей системы | End-to-end сценарии |
| **Приёмочное** | Проверка соответствия требованиям | UAT сценарии |

#### 4.1.7 По автоматизации
| Метод | Описание | Пример |
|-------|----------|--------|
| **Автоматическое** | Тесты запускаются скриптами | `python manage.py test` |
| **Ручное** | Тестирование вручную | UI/UX проверки |

### 4.2 Инструменты тестирования

| Инструмент | Назначение |
|------------|------------|
| **Django Test Framework** | Автоматизированные тесты |
| **unittest** | Модульные тесты |
| **coverage.py** | Измерение покрытия кода |
| **flake8** | Статический анализ кода |
| **Postman** | Тестирование API (будущее) |
| **Selenium** | UI-тестирование (будущее) |

## 5. Критерии тестирования

### 5.1 Критерии прохождения тестов
| Критерий | Значение |
|----------|----------|
| **Покрытие кода** | ≥70% |
| **Прохождение тестов** | 100% (все тесты должны проходить) |
| **Критические баги** | 0 (должны быть исправлены) |
| **Серьёзные баги** | ≤3 (допускаются) |
| **Мелкие баги** | ≤10 (допускаются) |

### 5.2 Критерии приостановления тестирования
| Причина | Действие |
|---------|----------|
| **Блокирующие баги** | Приостановить, исправить, продолжить |
| **Сбой инфраструктуры** | Приостановить до восстановления |
| **Изменение требований** | Пересмотреть тест-план |

### 5.3 Критерии завершения тестирования
| Критерий | Значение |
|----------|----------|
| **Все тесты выполнены** | Да |
| **Все критические баги исправлены** | Да |
| **Покрытие кода ≥70%** | Да |
| **Документация обновлена** | Да |
